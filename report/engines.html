At C=1 they tie (llama.cpp 61 ≈ vLLM 68 > SGLang 35). As concurrency climbs, vLLM (PagedAttention) and SGLang (RadixAttention) scale near-linearly while llama.cpp plateaus and goes host-bound (382→268→421). By C=64 vLLM/SGLang are ~4× llama.cpp, with vLLM also holding the lowest TTFT under load.

Which for whatllama.cpp: simplest to deploy (one static binary, GGUF, CPU/Metal reach), competitive at low concurrency, higher memory floor (preallocated KV). vLLM: throughput+latency winner for many users, paged KV — but a heavier, version-sensitive stack (as the VOID episode showed). SGLang: scales like vLLM under load (RadixAttention shines with shared prefixes — which we disabled, so its multi-turn edge is understated), slowest at C=1.
Keep attachedThis is bf16 for all three — not each engine's optimal quant (low-bit GGUF / AWQ / FP8 would shift their own curves); prefix caching off; one 4B model / one workload. No engine is "inherently" fastest — the numbers combine scheduler, KV design, and kernels. The new models on the model-scaling page ran on llama.cpp only (vLLM 0.11.0 predates their architectures).

What a real chatbot needs (mostly 2025–2026 literature; prefix caching now measured on this rig — see Context & accuracy)

Prefix caching is the top lever for multi-turn

Real chats are multi-turn: the system prompt + history is a huge shared prefix, each new turn a tiny suffix. With prefix caching (automatic in vLLM/SGLang) only the uncached suffix is prefilled. Our E3 deliberately disabled it (--no-cache-prompt) — measuring the worst case (4.3∶1 prefill∶decode). Our §9 chatbot A/B then measured it directly: at 2–4k depth prefix caching cut TTFT ~6.8× (505→75 ms), and the ON/OFF trends cross (ON falls with depth, OFF rises). At scale, one multi-turn KV-reuse system cuts P99 TTFT up to −69% vs vLLM/SGLang KV-cache baselines (SwiftCache). Enable it before tuning anything else.

TTFT & TPOT beat aggregate throughput for UX

TTFT = time to first token; TPOT = inter-token latency (streaming smoothness). At our throughput peak (C=64) aggregate tok/s is maximal but each user waits ~20 s. The production metric is goodput — requests meeting a latency SLO — not raw throughput. Pick your operating point from an SLO.

Use vLLM / SGLang when…

  • many concurrent users on a GPU, latency SLAs
  • you want PagedAttention + native continuous batching + prefix caching + speculative decoding
  • native multi-LoRA (many adapters, one base)

Use llama.cpp when…

  • single-user desktop / edge / offline
  • a quick prototype or model-swapping via GGUF
  • CPU or mixed hardware, trivial deployment
Context for the scary numbersA Red Hat test cites ~44× tok/s for vLLM — but that used Llama-3.1-8B full-precision on one H200 (a very different config from our 4B/bf16/A5000), and the source calls it config-specific. Our own measured gap here is ~4× at C=64, in the same direction. Config caveat: SGLang ran with CUDA graphs disabled (startup robustness) while vLLM used them — so this is iso-precision, not iso-configuration, and SGLang is somewhat handicapped.

Speed vs accuracy, and the latest levers

Quantization: Q6_K near-lossless; at 4-bit, GGUF-Q4/AWQ/GPTQ are close and model/task/calibration-dependent (AWQ often matches/beats GPTQ per its paper, ~1.45× kernel speedup). FP8: the A5000 lacks native FP8 W8A8 acceleration (needs Ada/Hopper), though Ampere runs FP8 weight-only via Marlin. The 2025–26 levers — speculative decoding, chunked prefill, prefix caching, disaggregated prefill/decode (vLLM still labels the last experimental) — typically move TTFT and effective concurrency more than a faster quant.

Bottom lineFor a single user or edge/CPU deploy, llama.cpp is the pragmatic pick (simplest, competitive at C=1). For many concurrent users, vLLM and SGLang are ~4× faster here and hold better tail latency — pick vLLM for the lightest-latency p50, SGLang if its RadixAttention prefix reuse fits your multi-turn workload (we disabled it, so its real edge is understated). These are bf16 numbers on one model — not a universal ranking.
