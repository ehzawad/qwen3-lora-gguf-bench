Precision knobs — decode is bandwidth-bound, prefill isn't

Two opposite lessons from the same sweep: lower-bit weights make decode faster (up to 2.18×), while prefill barely cares about bit-width.

These are single-stream llama-bench micro-benchmarks — raw kernel throughput (pp512 = prefill, tg128 = decode; no HTTP, no sampling, no real prompts). A different quantity from the server numbers in the concurrency study — don't cross the streams. Caveat: 3 samples/point, the first a cold outlier — read prefill as "flat," not a fine ordering.

Decode tok/s falls monotonically with weight size

Q4_K_M (2.32 GB) → bf16 (7.49 GB). Decode reads every weight per token ⇒ bandwidth-bound.

  • KV dtype is a memory knob, barely a speed knob. q8_0 KV halves the footprint for only ~4% decode cost — worth it only when KV memory is the real constraint (long context / many slots), not at short single-stream. bf16 KV showed no advantage here.
  • Flash-attention: on (this build/model). +15–21% prefill, +4–9% decode — effectively free here.
Speed vs accuracy — now measured on this rigQ4_K_M's 2.18× decode win carries a quality cost we now measure directly: a full-test wikitext perplexity sweep of exactly these five quants puts Q6_K/Q8_0 practically indistinguishable from bf16 (≤0.014% PPL, no paired significance test) and Q4_K_M at +2.42% PPL. That is general-domain likelihood, not legal-task accuracy — and cross-scheme rankings stay literature (AWQ often matches/beats GPTQ per its paper, ~1.45× kernel speedup). Validate quality on your task before dropping bits. See Context & accuracy for the numbers, Engines for the FP8/AWQ engine angle.
