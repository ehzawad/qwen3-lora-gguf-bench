LLM Inference on a Single A5000

A numbers-first field guide to what actually runs out first when you serve a small fine-tuned model from one GPU. In the short-chat regime it is not VRAM — not even the GPU's compute. Open the context window and keep sessions resident and it flips to VRAM-bound (Context & accuracy).

A 4B model at Q6_K is ~3.3 GB — tiny against 24 GB. So the question was never "does it fit?" It's "how many users, how fast, and what breaks first?" Most of what follows is measured on one rig (§6 adds literature-sourced nuances, cited); the caveats in §8 are the difference between seeing a number and understanding it. Two claims are flagged loudly throughout: the vLLM engine comparison did not happen (it crashed at warmup), and the "extract a LoRA with SVD" probe measures something subtler than its nickname.

§1

Concurrency capacity — the core finding

Method (label it precisely): llama.cpp, closed-loop barrier-synchronous load, ignore_eos, ~242-token prompts (231–253) / exactly 256 generated, --parallel == C, 768-token slots. This is a saturation-capacity probe, not a real arrival-pattern test — keep that for §6.

Aggregate throughput climbs to a broad plateau over C = 32–64, peaks at C = 64 (801.8 tok/s), then declines to 749 by C = 128. There is no simple "knee at 30": the biggest single batching win is C8→C16 (+262 tok/s), and marginal gain goes negative past 64.

The per-user view collapses the whole way up

Fair-share throughput (aggregate ÷ C) falls from the first step: 118 → 12.5 at the peak → 5.9 tok/s at C=128 (a 20× degradation). Because each request emits a fixed 256 tokens, latency is just its inverse: p50 climbs 2.2 s → 43.6 s. At the throughput peak, each user already waits ~20 s. "Max throughput" and "good latency" are different operating points.

What actually breaks first: the bottleneck leaves the GPU

This is the counterintuitive core. As C rises, the GPU does less (util median 94%→41%, power 229 W→168 W) while the llama-server process CPU rises ~11× (2.5%→29%). The GPU isn't idle — it's bursty/starved: p95 util pins at 99% while the median sits at ~40%, a distribution consistent with gaps between batches. The evidence most strongly points to a host-side feed/orchestration bottleneck (CPU-scheduling overhead the leading candidate); we did not profile to isolate the exact mechanism.

Measurement hygienenvidia-smi utilization.memory is the fraction of time the memory controller was busy — a proxy, not GB/s. It falls at high C simply because the GPU idles more between bursts. No bandwidth claim rests on it.

§2

Precision knobs — decode is bandwidth-bound, prefill is not

Single-stream llama-bench micro-benchmarks (raw kernel throughput; a different quantity from the server numbers — don't cross the streams). Decode reads every weight once per token, so fewer bits = faster decode. Prefill is a big batched GEMM, so it's compute-bound and flat.

  • Flash-attention: on (this build/model). +15–21% prefill, +4–9% decode — effectively free here.
  • KV-cache dtype is a memory knob, barely a speed knob. q8_0 KV halves the KV footprint for ~4% decode cost — worth it only when KV memory is the real constraint (long context / many slots), not at short single-stream.
  • Choose weight quant by budget: Q4_K_M if decode speed rules; Q6_K/Q8_0 if quality rules. Prefill barely cares about bit-width.

Caveat: 3 samples/point, first a cold outlier — the ~17% prefill spread is within noise, reinforcing "flat," not any real ordering.

§3

Engine choice — and why this comparison is void

Do not violate thisThe llama.cpp-vs-vLLM comparison did not happen. vLLM 0.25.1 loaded the model, then EngineCore died at kernel_warmup with a Triton JIT parse bug — 0 requests served. No winner, ratio, or latency delta between engines can be stated; the hypothesis is untested, not refuted. And it wouldn't have been iso-precision anyway (llama.cpp Q6_K 6-bit vs vLLM bf16 16-bit).

The llama.cpp side did run cleanly on a fixed 100-slot server (below). It independently reproduces §1: sublinear scaling (6.4× for 100× clients), sharp tail-latency growth, and the same bursty/host-bound signature (GPU util 93%→41% while p95 pins at 99%). Note C=30 here is only 370 tok/s — half of §1's dedicated-30-slot 720. This cross-run difference is real and consistent with over-provisioning overhead (70 idle slots at C=30), though the exact cause wasn't isolated — a reason to right-size slots to expected load.

§4

Is a LoRA "just a matmul"? The SVD-extraction probe

Mechanically, yes: a merged LoRA is W + (α/r)·B·A — base weights plus a low-rank matmul. The tempting leap is "so I can subtract any two checkpoints, SVD the difference, and recover the LoRA that made it." The study tests exactly that, with a control (a delta we know is a rank-16 LoRA) and a treatment (the delta between two published checkpoints).

Read this framing, don't skip it
  • It is an inter-checkpoint delta, not a proven full fine-tune — Instruct-2507 declares no base model.
  • Any rank-r SVD is "LoRA-representable," never "a working extracted LoRA" — no task behavior was trained or validated.
  • The high-rank result does not refute "LoRA can match full fine-tuning" (cf. Thinking Machines, LoRA Without Regret): a trained low-rank adapter stores task information, not the full weight delta — orthogonal to this measurement.
Takeaway: "merged LoRA = base + low-rank matmul" is true; "so I can SVD any diff back into a LoRA" is false in general — the real between-checkpoint delta is high-rank, embedding-shifting, and MLP-heavy.
§5

Synthetic finite-user load (Locust) — closed-loop, not real traffic

More traffic-like than the barrier sweep, but not real traffic: a finite pool of users with think-time between requests. Still closed-loop — user count is an upper bound on in-flight concurrency, and offered load is throttled by the pool, not pushed to saturation. Never compare these tok/s with the §1 saturation numbers; they measure different worlds.

Always ask: expensive in what units, and is that the bottleneck?

§6

Nuances — what a real chatbot needs, and llama.cpp vs vLLM

Everything above measured raw serving capacity on one engine. A production chatbot cares about different things. This section is grounded in the 2025–2026 literature (cited), not our own measurement.

  • Prefix caching is the #1 chatbot lever. Real chats are multi-turn: the system prompt + history is a huge shared prefix, each new turn a tiny suffix. With prefix caching (automatic in vLLM/SGLang) only the uncached suffix is prefilled. Our E3 deliberately disabled it; our §9 chatbot A/B then measured it directly — at 2–4k depth prefix caching cut TTFT ~6.8× (505→75 ms), and the trends cross (ON falls with depth, OFF rises). For scale, one multi-turn KV-reuse system cuts P99 TTFT up to −69% vs vLLM/SGLang KV-cache baselines (SwiftCache).
  • TTFT & TPOT beat aggregate throughput for UX. At our peak (C=64) aggregate tok/s is maximal but each user waits ~20 s. The production metric is goodput (requests meeting a latency SLO), not raw throughput.
  • llama.cpp vs vLLM — with a caveat on the scary numbers. vLLM (PagedAttention + native continuous batching + prefix caching + speculative decoding) is built for many concurrent users; a Red Hat test cites ~44× tok/s and stable sub-second TTFT at 64 users vs a llama.cpp config whose TTFT exceeded 180 s — but that test used Llama-3.1-8B at full precision on one H200 (very different from our 4B/Q6_K/A5000), and the source calls it workload-specific, not a general verdict. The gap is config-dependent: our -cb, --parallel==C llama.cpp had p99 TTFT ~7.5 s at C=128, yet still hit a host-bound wall (peak then decline). vLLM's scheduler + PagedAttention are designed to improve high-concurrency serving; whether vLLM would push past this rig's wall is unmeasured — it crashed at warmup. No engine winner is stated.
  • Speed vs accuracy — now measured (Context & accuracy). A full-test wikitext perplexity sweep of the same merged model shows Q6_K and Q8_0 practically indistinguishable from bf16 (≤0.014% PPL, each carrying its own ~±0.076 run-level uncertainty; no paired significance test computed) and Q4_K_M at +2.42% PPL for 2.18× decode. But that's general-domain next-token likelihood, not legal-task accuracy, and the bf16 GGUF is a same-precision reference, not lossless ground truth — so "validate on your task" still stands. At 4-bit the AWQ paper reports AWQ often matches/beats GPTQ in its settings (~1.45× kernel speedup). FP8: the A5000 lacks native FP8 W8A8 acceleration (needs Ada/Hopper), though Ampere runs FP8 weight-only via Marlin.
  • 2025–26 levers that matter more than kernel speed: speculative decoding, chunked prefill, prefix caching, and disaggregated prefill/decode — mostly mature in vLLM/SGLang; llama.cpp has speculative decoding + slot prompt-cache, not the full suite.
Bottom line for a chatbot: our numbers are the floor — one engine, no prefix cache, no speculative decoding. On the right engine with prefix caching on, a multi-turn chatbot has far lower TTFT under load and higher usable concurrency. Production multi-user chatbot → vLLM/SGLang; single-user, edge, prototype, or CPU/mixed → llama.cpp.

§8

Limitations you must carry with the numbers

  • One GPU, one build, one model. A5000 24 GB, llama.cpp 91d2fc38, Qwen3-4B Q6_K. The peak at 64, the 108 MiB/slot slope, the CPU crossover — all rig-specific; they don't transfer.
  • Closed-loop ≠ open-loop. §1 is a fully-backlogged saturation probe; Locust is finite-user think-time. Their tok/s are not comparable and neither is an open-loop Poisson-arrival SLO test.
  • utilization.memory is a proxy, not bandwidth (GB/s). And GPU util median is misleading alone — reason from the median-vs-p95 gap (median 40% with p95 99% = bursty/starved).
  • The engine comparison is void — vLLM served 0 requests. Every E5 number is llama.cpp-only.
  • The SVD study measures an inter-checkpoint delta, "LoRA-representable" ≠ a validated adapter, and it refutes nothing about trained-LoRA quality. bf16 read-back also inflates the control's rank floors.
  • E4 is 3 samples/point (first a cold outlier); "prefill is flat" means no monotonic bit-width ordering — n=3 is too thin for a robust ranking.
  • We measured speed, memory, and wikitext perplexity — not legal-task quality. A full-test (583-chunk) PPL sweep of all five precisions (Context & accuracy) found Q6_K/Q8_0 practically indistinguishable from bf16 (≤0.014%, no paired significance test) and Q4_K_M at +2.42% PPL — but that is general-domain next-token likelihood, not contract-intake accuracy, and cross-scheme/AWQ/GPTQ claims (§6 D) remain external literature. Validate quality on your task before dropping bits.
  • The host-bound diagnosis is an inference, not an isolated cause — no profiler/CUDA trace/thread ablation was run. Manifests also record source_dirty=true; raw per-slot server logs are gitignored (derived evidence is committed).
