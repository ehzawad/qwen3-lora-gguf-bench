Context window, concurrency & the accuracy↔speed tradeoff

Open the context window and keep sessions resident and the whole story inverts: the short-chat regime was compute/host-bound, this one is VRAM-bound. Capacity is now set by KV bytes, and every deep token decodes slower.

1 · Context window is the VRAM constraint

The KV cache is linear in tokens, and on this A5000 the per-token cost is measured, not guessed — derived straight from the boot-allocation frontier: (22226−4916) MiB ÷ (30720×4 tok) = 144.25 KiB/token for f16. Note q8_0 KV lands at 53% of f16 (76.5 KiB/tok), not a clean half — 8-bit KV still carries per-block scale overhead. You don't get a free doubling.

Read this carefullyq8_0 nearly doubled resident slots (16→30) at similar VRAM, but aggregate throughput barely moved (135→113) while fair-share halved — a pure latency-for-capacity trade. "3.78" is fair-share (agg/30), not a measured per-request rate; the measured median per-request decode was 5.7 tok/s — still below comfortable reading speed. And the run was not clean: 3 of 90 follow-up turns grew past 8192 tokens and were rejected, so this proves memory feasibility, not that every session stays in-window. The brutal p95/max TTFT is a zero-think-time closed loop (every slot always has a request queued) — no arrival process, so it does not estimate open-loop production latency. 100 residents blow the KV budget outright. Note: q8_0 KV's accuracy cost is unmeasured here — §4's perplexity sweep quantized the weights, not the KV cache.

3 · Prefix caching flips the TTFT trend

If you take one operational lever from this page, take this. With prefix caching on, deeper history raises the cache-hit rate and lowers TTFT; with it off, every turn reprocesses the whole prompt and TTFT climbs. Decode rate is untouched (~99–104 tok/s either way) — caching is purely a prefill/TTFT effect. Measured over 150 scripted-user turns (model generates the replies) replaying individual UltraChat conversations at server ctx 32768:

For any multi-turn deployment this is non-optional — turning it off makes conversations get slower the longer they run. (Deep buckets have n=2; indicative, not robust.)

4 · The accuracy↔speed tradeoff — measured

This time accuracy is measured, not borrowed. All five runs are the same merged model at different GGUF precisions, scored on all 583 disjoint 512-token chunks of the wikitext-2-raw test using llama-perplexity's default half-window scoring (each chunk scores its second 256 tokens — not strided sliding-window; no --ppl-stride), paired with E4 decode tok/s on the same A5000.

Speed rises monotonically as bits drop (77→168 tok/s). PPL does not: the point estimates are practically flat through Q6_K (bf16/Q8/Q6 within 0.014% of each other, each carrying ~±0.076 run-level uncertainty — bigger than the gaps), then rise clearly at Q5 (+1.50%) and Q4 (+2.42%). Because every model scored identical text, a real significance test needs a paired per-chunk loss-difference analysis, which we did not compute — so this is not a formal "within-noise" or "lossless" claim. Honestly: Q6_K/Q8_0 are practically indistinguishable from bf16 on this corpus (≤0.014% delta), so Q6_K is the sensible default; Q4_K_M costs +2.42% PPL to buy 2.18× decode (and its small file leaves more room for KV — the real bottleneck per §1).

Before you quote theseThis is general-domain next-token likelihood on wikitext, not legal-task accuracy — lower PPL here does not certify contract-intake quality. The bf16 GGUF is a same-precision reference (ΔPPL ≡ 0 by definition), not lossless ground truth vs the original adapter model. "+2.42% PPL" is not "Q4 loses 2.4% quality" and says nothing about LoRA fidelity.

Which LoRA

Every number on this page is one model: narcolepticchicken/qwen3-4b-legal-ops-contract-intake-lora (Qwen3-4B, LoRA rank 16 / alpha 32), merged into the base weights before GGUF export. So there is no adapter-swap or separate-adapter overhead in any measurement — a single merged checkpoint at five precisions, and the KV/context frontier is set by the 4B base geometry, not by the adapter.
