LoRA → GGUF, and "is a LoRA just a matmul?"

Yes, mechanically — a merged LoRA is W + (α/r)·B·A. But "so I can SVD any two checkpoints back into the LoRA that made them" is false in general. Here's the receipt.

  • The control behaves low-rank — eff-rank ~43, embeddings/norms delta exactly 0. (It isn't a clean "16" only because it's read back through bf16-merged weights, so bf16 rounding smears energy across the spectrum — a known calibration artifact.)
  • The real delta is high-rank — rank-16 keeps only 3.55% of the energy; even rank-512 keeps under half. It shifts embeddings ~20% (a target-linear LoRA can't represent that at all), and MLP is harder than attention.
Read this framing, don't skip it
  • It's an inter-checkpoint delta, not a proven full fine-tune — Instruct-2507 declares no base model.
  • Any rank-r SVD is "LoRA-representable," never "a working extracted LoRA" — no task behavior was trained or validated.
  • The high-rank result does not refute "LoRA can match full fine-tuning" (cf. Thinking Machines, LoRA Without Regret): a trained low-rank adapter stores task information, not the full weight delta — orthogonal to this measurement.
Takeaway: "merged LoRA = base + low-rank matmul" is true; "so I can SVD any diff back into a LoRA" is false in general — the real between-checkpoint delta is high-rank, embedding-shifting, and MLP-heavy.
