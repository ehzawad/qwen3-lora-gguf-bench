Synthetic finite-user load & context growth

Closer to real traffic than the barrier sweep — but still closed-loop (a finite user pool with think-time). Never compare these tok/s with the saturation numbers; they measure different worlds.

E2 — short-chat, fixed 64-slot server

Independent short prompts, think-time 1–4 s, natural sampling (temp 0.7, max 160 tok), ramped users. Steady-state token throughput counted only inside a declared window. 0 failures.

E3 — multi-turn growing context (20 slots, 4096 tok)

Each user grows a real conversation up to 8 turns, then resets. Prompt caching was deliberately disabled (--no-cache-prompt) — so the whole history is re-prefilled every turn (the worst case). Overall 205.9 tok/s, 4.3∶1 prefill∶decode token ratio, 0 failures.

The teaching caseThe scary metric (4.3∶1 prefill∶decode ratio) turns out cheap in seconds: prefill runs ~3,100 tok/s overall (up to ~4,200 at the longest turns) while decode runs ~13 tok/s effective, so re-prefill costs only ~40 → ~300 ms of TTFT and barely moves the ~10 s decode-dominated e2e. Always ask: expensive in what units, and is that the bottleneck? (In a real chatbot you'd turn prefix caching on — see Engines & chatbot — and even this small tax largely vanishes.)
