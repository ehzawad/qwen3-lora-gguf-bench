## A. Fair-comparison rules

### Claims you may make

Use language like:

> “On one RTX A5000, with llama.cpp commit X and configuration Y, these exact text-only Q4_K_M GGUF artifacts achieved the following native-token throughput, latency, and measured memory footprints.”

You may compare:

- Single-stream prefill and decode.
- Throughput/latency at the same concurrency and sequence lengths.
- Scaling curves and measured full-GPU feasibility.
- Static VRAM, KV growth, workspace peaks, and the measured context × concurrency frontier.
- Each model’s peak, provided you also show the shared feasible concurrency domain.

Rerun the 4B baseline with the new llama.cpp binary. Its old-vs-new result is a regression bridge; historical numbers cannot appear as the direct baseline in the new headline chart.

### Claims you must not make

Do not claim:

- “Iso-precision,” “equal 4-bit precision,” or equal retained quality. Say **matched nominal Q4_K_M tier**.
- That one architecture is inherently faster or more memory-efficient. Results combine architecture, artifact recipe, kernels, tokenizer, batching, and runtime support.
- That native tok/s represents equal text or useful work across tokenizers.
- That a larger KV frontier means better context capability; attention semantics may differ.
- Any accuracy, legal quality, multimodal, or universal performance ranking.
- That Qwen performs 9.65B parameters of computation per token.
- That Gemma is a “2B model,” uses 2B parameters per token, or should have 2B-class VRAM.

Report Qwen as **~9.65B total parameters, MoE**. Report active parameters or routing topology only after authoritative/runtime verification. Active parameters do not determine resident expert memory.

Report Gemma as **~5.1B total parameters, E2B designation (~2B “effective” as supplied)**. E2B/MatFormer/per-layer-embedding terminology does not establish what current llama.cpp executes or predict decode speed or VRAM; verify before interpreting it.

### Quantization confound

Reconciled approach:

1. Use prebuilt GGUFs for bring-up.
2. Preferred headline set: self-quantize all three from verified bf16 sources with one pinned converter/quantizer, identical Q4_K_M flags, and one recorded imatrix/calibration policy.
3. Audit tensor types, exclusions, tensor bytes, and effective bytes/parameter. Identical commands still may not produce identical tensor treatment across architectures.
4. If conversion support is unreliable, publish the prebuilt set as an **“available-artifact comparison”**. Do not imply controlled precision.
5. Do not mix self-quantized and third-party artifacts in one controlled headline ranking.

Use two workload views:

- **Sequence-length controlled:** exactly 256 post-template native input tokens and 256 generated tokens.
- **Application cross-check:** identical raw messages through each native template; report actual token counts, request latency, requests/s, and optionally output characters/bytes per second.

Neither view controls both semantics and token count.

### Provenance and execution

- New llama.cpp commit, executable hash, compiler/CUDA versions, build flags.
- Old and new 4B bridge result.
- Exact GGUF repository revision, filename, SHA-256, file size, GGUF metadata, tensor-type histogram, and quantization logs where available.
- Server and `llama-bench` flags: GPU layers, threads, context, batch/ubatch, KV dtype, flash attention, scheduler and prefix-cache settings.

- Warm-up, repetitions, run ordering, and dispersion—not only the best run.

### Placement and memory

- Full-GPU versus CPU-offloaded tensors, including expert placement.
- Load/static VRAM, host RAM, runtime/workspace peak, KV allocation, and first OOM/failing point.
- Never silently reduce GPU layers, batch size, or other flags for one model.
- Require successful completion, not merely server launch, when defining the frontier.

### Performance

- Separate prefill from batch-1 decode.
- At every concurrency: native output tok/s, requests/s, TTFT p50/p95, inter-token or end-to-end latency, actual input/output lengths, failures, and OOMs.
- Sweep requested C=1..100 until safe failure. Show:

  - Comparisons at identical concurrency in the shared feasible domain.
  - Each model’s independently labeled peak and maximum feasible point.

- Total parameters and resident weight bytes.
- Expert count, routing/top-k, and active-parameter estimate only if verified.
- Actual/effective internal batch size, if exposed.
- C=1 behavior and the complete concurrency curve.
- GPU and host utilization.
- Repeated runs over a fixed, diverse, unique prompt set.

### KV and tokenizer correctness

- Runtime-verified layer count, KV heads, head dimension, attention modes, and sliding-window behavior; do not fill these from memory.
- Theoretical KV bytes/token as a cross-check, plus empirical VRAM slope over context and live sequences.
- Separate allocated KV from live/effective KV where exposed.
- Empirically test whether any sliding-window implementation reduces or plateaus allocation.
- Tokenizer revision, embedded chat template, BOS/EOS handling, and server-observed post-template counts.
- Confirm no projector is loaded and no media inputs are used. Verify that text-only serving works without missing-projector or unsupported-tensor warnings.

## C. Ranked risks and mitigations

1. **Incomplete or incorrect new-architecture support.**  
   Pin one tested master commit. Gate on clean loading, metadata parsing, reference-tokenizer parity, deterministic generation, exact lengths, and no unsupported/missing-tensor warnings. Otherwise report “unsupported at tested commit.”

2. **Build drift invalidates the baseline.**  
   Run all three on the same new binary; retain the old binary and use the 4B old/new bridge only to quantify drift.

3. **Quant recipes masquerade as architecture differences.**  
   Prefer the controlled self-quantized set. Otherwise downgrade to exact-artifact claims and publish the full quant audit.

4. **Tokenizer/template mismatch invalidates 256-token fairness.**  
   Count after template application using the server tokenizer; retain the identical-raw-message cross-check.

5. **Silent CPU offload or 24 GB exhaustion.**  
   Preflight load, short generation, then context/concurrency ramps. Inspect placement logs, VRAM, and host RAM. Mark failures rather than changing placement.

6. **Unverified KV/sliding-window implementation behavior.**  
   Measure allocation and decode behavior empirically. Do not infer a memory advantage from the architecture label.

7. **MoE and host-side batching make curves workload-specific.**  
   Use diverse unique prompts, repeat runs, expose actual batching where possible, and report latency alongside throughput.
